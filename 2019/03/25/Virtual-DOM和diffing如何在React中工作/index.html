Virtual-DOM和diffing如何在React中工作

发表于 2019-03-25   |   分类于 技术   |     |   阅读次数

工作中使用react的时间蛮久了，其实一直很想了解虚拟Dom的工作原理，虽然使用久了，在表象上它看起来跟上面图一样蛮清晰的，但是终究感觉有点不得劲。所以需要再深入的了解下。

在这之前，我不知道大家曾经或者此刻有没有想过一个问题，那就是为什么react中会使用virtual Dom呢？

如果有人在这之前问我上面的问题，我肯定会耐心的（不耐烦的）告诉他，因为“操作dom比较慢”。这个回答其实看上去蛮能敷衍每个提问者的，因为我们毕竟知道了原因。不过我们还是先简单看看Dom的构建流程，然后再回到这个上面问题。


这个图在我们初入前端时，应该都有了解过，它展示了html转换成Dom到绘制到屏幕的整个过程。每次Dom的某些内容发生变化的时候，由于Dom被表示为一个树结构，因此对Dom的更改是非常快的，但是更改可以理解为只是在代码层面（跟你知道vdom操作一个样），而更改后的元素，必须经过上面图中重排阶段，然后把新的结果重绘到界面，这个过程是非常慢的，有时侯慢的可能你就感知到了，而且只要你在这过程中再进行一次DOM更新，整个渲染流程都会重做一遍因此，重排/重绘的项目越多，我们的应用就越慢。
而刚好 Virtual DOM试图最小化 重排/重 绘这两个阶段，从而为大型复杂应用程序获得更好的性能
每当你想对视图进行一次更新，实质上那些本该直接作用于真实DOM的改动，都会先作用于Virtual DOM，然后再将要改动的部分通知到真实DOM。这样可以大幅减少DOM操作带来的重计算步骤。

对于DOM 操作 真正被诟病的是每次操作都会触发布局的改变、DOM树的修改和渲染。假设我们去修改二十多个节点，一个一个去修改，就会引起20多次潜在的布局重排和重绘。而采用virtual Dom则不会（尽可能少的）区去重新渲染，它会将你这个批次的修改尽可能一起去更新到真实Dom，这样上面二十多个节点的改动就会变成一次重排和重绘。当然，我们也可以不用virtual Dom，而自己手动一次性更改，也能达到这样的结果。

了解diff

React 中最值得称道的部分应该就是 Virtual DOM 与 diff 的结合，特别是其高效的 diff 算法，让用户可以无需顾忌性能问题而”任性自由”的刷新页面，让开发者也可以无需关心 Virtual DOM 背后的运作原理，因为 React diff 会帮助我们计算出 Virtual DOM 中真正变化的部分，并只针对该部分进行实际 DOM 操作，而非重新渲染整个页面，从而保证了每次操作更新后页面的高效渲染，因此 Virtual DOM 与 diff 是保证 React 性能口碑的幕后推手。

diff 策略
1、Web UI 中 DOM 节点跨层级的移动操作特别少，可以忽略不计。

2、拥有相同类的两个组件将会生成相似的树形结构，拥有不同类的两个组件将会生成不同的树形结构。

3、对于同一层级的一组子节点，它们可以通过唯一 id 进行区分。

以上三个策略，React 分别对 tree diff、component diff 以及 element diff 进行算法优化，事实也证明这三个前提策略是合理且准确的，它保证了整体界面构建的性能。

reconciler（调和）

diff 算法和 virtual dom 的完美结合的过程被称为 reconciler，有了 reconciler，开发者可以脱身操作真实的 dom 树，只需要向 react 描述界面的状态，而 react 会帮助你高效的完成真正 dom 操作。
在 react16新版中对reconciler做了重大改变，采用了fiber架构来做reconciler,从而实现了异步渲染。原来的reconciler过程是一个同步递归过程，当状态改变的时候，会从根节点开始遍历，找出所有的不同，而对于特别庞大的 dom 树来说，这个递归遍历的过程会消耗特别长的时间。在这个期间，任何交互和渲染都会被阻塞，这样就给用户一种“死机”的感觉。而fiber把 reconciler 的过程拆分成了一个个的小任务，并在完成了小任务之后暂停执行 js 代码，然后检查是否有需要更新的内容和需要响应的事件，做出相应的处理后再继续执行 js 代码。这样就给了用户一种应用一直在运行的感觉，提高了用户的体验。当然他对每个拆分的小任务做了优先级标记，这样保证了重要的更新能及时同步在页面中。

具体原理大家可以查看相应的源码或者文章，不再引申，这里只是抛出fiber这个概念。
